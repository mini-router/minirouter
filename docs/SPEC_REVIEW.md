## (B) Internal contradictions / numbers that don't add up

- [ ] **λ recomputation is arithmetically WRONG.** Spec §5.3 and §9 both claim `⌈4 + 3 ln 15360⌉ = 34`. Actual: `4 + 3·ln(15360) = 32.919 → ⌈⌉ = 33`, **not 34**. This propagates everywhere (population size, budget). Fix to **λ=33**.
- [ ] **Budget arithmetic WRONG twice over.** §5.3 row says `B_env ≈ m_CMA·λ·T = 16·34·60 = 34,560`. But `16·34·60 = 32,640`, not 34,560. And with the *correct* λ=33 it's `16·33·60 = 31,680`. The "34,560" figure appears in two places and is internally inconsistent with its own formula. Both the multiplier (34 vs 33) and the product (34,560 vs 32,640) are errors.
- [ ] **The 9,216 SVF count is suspicious and likely back-fitted.** `9,216 = 9 × 1024`. For SVF, a learnable scale exists *per singular value*, and a matrix of shape `m×n` has `min(m,n)` singular values. For Qwen3-0.6B (d=1024) the attn Q/K/V/O are ~1024×1024 → 1024 SVs each (4×1024=4096); MLP matrices are 1024×D_ff (D_ff≫1024) → still 1024 SVs each. So "9 matrices each contributing exactly 1024 SVs" *can* reach 9216, but Qwen3 uses GQA (K/V projections are smaller, e.g. 1024×512 → 512 SVs) and SwiGLU (3 MLP matrices: gate/up/down). The spec's "Q/K/V/O + gate/up/down = 7 matrices" gives **7×1024=7168 ≠ 9216**, not 9. The spec even admits "if the exact matrix set can't hit 9,216, fall back to all-linear, record actual count" — meaning **the 9,216 figure may be unreproducible with the stated matrix set**. This is a self-contradiction the spec papers over.
- [ ] **Head param-count claim "~10K" vs reality.** Spec §1.1 says "~10K-param linear head" and "<20K trainable." For OURS the head is 6,144 (correct: 6×1024). Fine — but §3.4 then calls theory's "n≈10000" as "loosely the head," while the optimized vector is 15,360. The spec optimizes 15,360 but repeatedly invokes paper hyperparameters derived for "n≈10000." It never resolves whether λ/c_cov should be computed on 15,360, 6,144 (head-only), or 10,000. The extraction flags this as an open decision; the spec silently picks 15,360 without justifying why paper-tuned constants transfer.
- [ ] **`m_0` initial mean contradicts the "uniform policy" claim.** §5.3 sets head `W=0` "→ uniform policy." With `W=0`, all logits are 0 → softmax uniform: correct. But SVF scales init at 1.0 means the SLM is *unmodified* at init, so `h` is fixed but the head outputs uniform regardless of `h`. That's fine, but note: with `W=0` the gradient/CMA signal through the head at iteration 0 is purely from σ-perturbations, and a 0-mean start with σ₀=0.1 on 15,360 dims gives logit perturbations ~N(0, 0.1²·||h||²) — if `||h||` is large (bf16 hidden states can have norm ~30–60), logits explode and softmax saturates immediately. σ₀=0.1 is asserted with no scale analysis. Risky, not contradictory, but the "uniform start" framing hides this.
- [ ] **REINFORCE batch size inconsistency.** §9 lists "batch=`m_CMA·λ`=512(paper)/544(ours)." With corrected λ=33, ours is `16·33=528`, not 544. The 544 came from the wrong λ=34.

## (D) Riskiest reproduction assumptions

- [ ] **#1 — SVF correctness.** Joint optimization of 9,216 SVF scales on the 2nd-to-last layer, with an unverified matrix set, is the highest-risk component. If the matrix set or SV count is wrong, θ layout is wrong, and *every* CMA result is invalid. And the "w/o SVF" ablation (R9) requires a working SVF to begin with.
- [ ] **#2 — Penultimate-token extraction matching train↔eval.** The paper's own ablation shows last-token vs penultimate is a >10pt swing on LCB. If the invented `<Head Input>` append procedure produces a different hidden state at train vs eval (e.g., tokenizer adds/strips a BOS, or generation vs single-forward changes the −2 index), the whole signal silently degrades. Must add an assertion test.
- [ ] **#3 — Paper CMA constants on a different n.** Assuming λ, c_cov, m_CMA=16 transfer from "n≈10000" to n=15,360 with weak parameter coupling unverified for OUR pool. The block-ε-separability that justifies sep-CMA-ES was measured on the paper's 7-agent representation space (SVM acc 0.99); there is zero evidence it holds for a 3-model Fireworks pool. R8 (CMA > others) could fail purely from this.
- [ ] **#4 — σ₀=0.1 with W=0 start.** No scale analysis of hidden-state norm vs perturbation magnitude. Either logits saturate (no learning) or stay near-uniform (slow). This single untuned number can sink a 60-iteration, thousands-of-API-call run.
- [ ] **#5 — Binary reward variance with m_CMA=16.** Fitness is a mean of 16 Bernoulli draws → std ≈ √(p(1−p)/16) ≈ 0.12 at p=0.5. CMA must rank λ=33 candidates whose true fitness differences may be <0.12. The optimizer may be ranking noise. The paper used the same, but with a richer 7-model action space; with 3 models the signal-per-candidate may be even flatter.
- [ ] **#6 — Caching keyed by prompt-hash silently breaks stochastic eval.** The spec adds a disk cache (model, prompt-hash, decode-params) AND uses temperature 0.0. With greedy decoding the same prompt always returns the same output, so "16 distinct instances" gives 16 *deterministic* trajectories per candidate — fine — but if any role prompt embeds the transcript verbatim, two CMA candidates that pick the same agent/role sequence will hit cache and get identical rewards, collapsing inter-candidate variance the optimizer needs. Interaction between caching + greedy + binary reward is unanalyzed.
- [ ] **#7 — Verifier-can-ACCEPT-on-turn-1 with empty solution.** The spec permits a turn-1 Verifier (free first role) and defaults unparseable verdicts to REVISE — but a Verifier on a near-empty transcript could spuriously ACCEPT, terminating with no answer → guaranteed reward 0. Early in training (uniform policy) this happens ~1/3 of Verifier-first trajectories, depressing the initial fitness floor and possibly biasing CMA away from ever selecting Verifier. No guard.
